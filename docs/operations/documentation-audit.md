# Аудит документации и план устранения технического долга (ноябрь 2025)

Документ фиксирует текущее состояние пользовательской и технической документации
ERNI-KI, выявленные разрывы и план их закрытия. Каноничный язык — русский;
обновления отражаются в `doc_version` и синхронизируются в локализации после
внедрения.

## Краткий вывод

- **Сильные стороны:** есть карта разделов в README, глобальная версия
  документации, пользовательский портал (RU/EN/DE), статус-страницы и базовые
  тренинги.
- **Основные разрывы:** неполный охват ролей (support/management), не хватает
  актуализированного HowTo каталога, нет единого стиля frontmatter и шаблонов
  для страниц, слабая связность между техническими и пользовательскими
  разделами, отсутствует регулярный цикл аудита/релизов документации.
- **Риски:** рост несогласованности между языками, устаревшие инструкции в
  тренингах/HowTo, потеря контекста при инцидентах из-за разнородных статусов.

## Методика оценки

- Проверена структура `docs/` и ссылки из README.
- Сверены frontmatter (`page_id`, `translation_status`, `doc_version`) и дата
  выпуска в `docs/VERSION.md`.
- Проанализированы ключевые порталы RU/EN/DE (`index`, `academy`, `status`).
- Оценены вспомогательные разделы: training, howto, news, operations/security.

## Наблюдения по разделам

- **README / карта документации:** ссылки есть, но нет явного входа в backlog
  документации и нет указания ответственных за обновления.
- **Academy (RU/EN/DE):** структура синхронизирована, но статус переводов в
  EN/DE требует регулярной сверки; нет явной связи с SLA по обновлению
  пользовательских гайдов при релизах платформы.
- **HowTo / training:** есть базовые материалы, но каталог не покрывает сценарии
  для разных ролей (поддержка, управление доступами, аудит логов), нет единого
  шаблона HowTo (цель/роли/шаги/ожидаемый результат).
- **Operations / security:** хорошие технические гайды, но отсутствуют короткие
  user-facing инструкции для инцидентов и FAQ для поддержки.
- **Версионирование:** глобальная версия внедрена, однако нет расписания
  регулярного аудита и не формализован триггер для инкремента версии при
  изменениях в HowTo/тренингах.

## Процесс регулярного аудита

- **Частота:** раз в месяц + после каждого релиза платформы.
- **Шаги:**
  1. Проверить актуальность README (карта, версии, контакты владельцев).
  2. Просмотреть Academy/порталы на трёх языках: статус переводов, актуальность
     ссылок.
  3. Перечитать все HowTo и тренинги: UI-скриншоты, команды, версии моделей.
  4. Обновить `docs/VERSION.md` и changelog документации при существенных
     правках.
  5. Задокументировать результаты аудита в этом файле и зафиксировать задачи в
     бэклоге.
- **Критерии выхода:** все P0 закрыты, обновлены статусы переводов, релизная
  заметка внесена в `docs/VERSION.md`.

## Следующие шаги (ближайшие 2 недели)

1. Подготовить шаблон HowTo и конвертировать текущие материалы.
2. Создать реестр владельцев разделов и закрепить ответственных.
3. Обновить политику версионирования с триггерами и включить её в чек-лист
   релиза.
4. Сформировать набор приоритетных HowTo для роли поддержки и добавить их в
   порталы.
