# Стратегия поддержки документации

Документ описывает процессы управления контентом ERNI-KI: кто отвечает за
обновления, как гарантируется актуальность и какие инструменты используются.

## 1. Цели

1. **Актуальность:** изменения в коде/инфраструктуре сопровождаются обновлением
   документации ≤ 24 часов.
2. **Точность:** отсутствуют битые ссылки, эмодзи, устаревшие версии.
3. **Доступность:** ключевые разделы доступны на RU/EN/DE.

## 2. Роли и зоны ответственности

| Роль              | Зона ответственности                                          |
| ----------------- | ------------------------------------------------------------- |
| Tech Writer       | Редактура, терминология, статус переводов                     |
| DevOps/SRE        | Runbooks, deployment/checklist, CI/CD описания                |
| Localization Lead | Координация EN/DE переводов                                   |
| Maintainer        | Запуск автоматических проверок, triage документационных issue |

## 3. Процесс обновления

1. **Инициирование:** PR с изменениями содержит ссылку на соответствующие файлы
   `docs/`.
2. **Подготовка:** обновляем Markdown + frontmatter (`last_updated`,
   `translation_status`).
3. **Проверка:** `pre-commit run --all-files`,
   `python scripts/docs/visuals_and_links_check.py`.
4. **Ревью:** минимум один approve от владельца раздела.
5. **Публикация:** после merge запускаются `mkdocs build` и link-check в CI.

## 4. Автоматизация

- `scripts/docs/update_status_snippet.py` — синхронизация статус-блоков в
  README, docs и локалях.
- GitHub Actions:
  - `ci.yml` — линтеры, mypy, тесты, link-check, docs metadata.
  - `docs-deploy.yml` — сборка MkDocs + GitHub Pages.
  - `update-status.yml` — автообновление статус-сниппетов при изменении
    `status.yml`.
  - `nightly-audit.yml` — ночной link-check/архив.
- pre-commit: markdownlint, no-emoji, detect-secrets, lychee (офлайн), status
  snippet check.

## 5. Расписание аудитов

| Частота       | Действия                                                     |
| ------------- | ------------------------------------------------------------ |
| Еженедельно   | Проверка link-check отчётов, исправление битых ссылок        |
| Ежемесячно    | Обновление `doc_version`, `last_updated`, проверка переводов |
| Ежеквартально | Комплексный аудит (см. `DOCUMENTATION_AUDIT_REPORT.md`)      |

## 6. Переводы

- `translation_status` отражает актуальное состояние
  (original/complete/pending).
- При изменении RU версии создаётся issue “Translation needed” с чек-листом.
- EN/DE файлы обязаны ссылаться на RU источник (см.
  `docs/en/reference/index.md`).

## 7. Версионирование и архивы

- Основная ветка документации — `develop`. Теги соответствуют релизам платформы.
- Исторические версии перемещаются в `docs/archive/**` с указанием даты и
  области.
- `docs/VERSION.md` содержит актуальный номер версии и правила обновления.

## 8. Качество и проверки

- PR не мерджится без успешного прохождения `ci` и `docs-deploy`.
- Любые временные placeholders запрещены.
- Проверки link-check/metadata/no-emoji являются блокирующими.
- Ownership разделов фиксируется в `docs/operations/core/admin-guide.md`.

Следование стратегии гарантирует, что документация синхронизирована с кодом,
переводы контролируются, а автоматизация предотвращает накопление технического
долга.
