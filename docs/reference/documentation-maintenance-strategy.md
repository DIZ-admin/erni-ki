# Стратегия поддержки документации

Документ определяет процессы обновления, проверки и версионирования всей
документации ERNI-KI.

## 1. Цели

1. Актуальность — все документы обновляются в течение 24 часов после изменений в
   инфраструктуре/коде.
2. Точность — отсутствуют битые ссылки, эмодзи, устаревшие версии.
3. Доступность — материалы доступны на RU/EN/DE, статус переводов отражён.

## 2. Ответственность

| Роль              | Ответственность                              |
| ----------------- | -------------------------------------------- |
| Tech Writer       | Обновление гайдлайнов, ревью контента        |
| DevOps/SRE        | Обновление runbooks, deployment/checklist    |
| Localization Lead | Координация переводов EN/DE                  |
| Maintainer        | Запуск автоматических проверок, triage issue |

## 3. Процесс обновления

1. **Инициирование.** Любое изменение в коде/инфраструктуре сопровождается
   ссылкой на соответствующий документ в PR.
2. **Подготовка.** Обновление Markdown-файла + статуса перевода.
3. **Проверка.** `pre-commit run --all-files` + `npm run docs:lint`
   (link-check).
4. **Ревью.** Минимум один APPROVE от владельца раздела.
5. **Деплой.** После мерджа запускается `mkdocs build` и CI link-check.

## 4. Автоматизация

- `scripts/docs/update_status_snippet.py` — синхронизация блоков статуса.
- `lychee` в CI — проверка ссылок в docs + README.
- pre-commit хуки: markdownlint, no-emoji, detect-secrets.
- `docs/reference/status.yml` — единый источник truth для статусных вставок.

## 5. Расписание аудитов

| Частота       | Деятельность                                                 |
| ------------- | ------------------------------------------------------------ |
| Еженедельно   | Проверка link-check отчётов, исправление битых ссылок        |
| Ежемесячно    | Обновление `doc_version`, `last_updated`, проверка переводов |
| Ежеквартально | Комплексный аудит (см. `DOCUMENTATION_AUDIT_REPORT.md`)      |

## 6. Работа с переводами

- Все ключевые разделы (operations, getting-started, security) поддерживаются на
  RU/EN/DE.
- При изменении оригинала создаётся issue «Translation needed» с чек-листом.
- `translation_status` меняется на `pending`, пока перевод не обновлён.

## 7. Хранение и версияция

- Источник — Git (ветка `develop`). Теги документов совпадают с релизами
  платформы (`v0.61.3`).
- Для крупных изменений создаётся changelog в `docs/reports/...`.
- Исторические версии сохраняются в `docs/archive/...` с указанием даты.

## 8. Контроль качества

- Проверки CI являются блокирующими (без успешного link-check, markdownlint и
  no-emoji PR не мерджится).
- Для сложных документов используется «документальные» review (2 ревьюера).
- Любые временные placeholder-документы запрещены.

Следование стратегии гарантирует, что документация ERNI-KI остаётся
синхронизированной с кодом и инфраструктурой и может быть использована
пользователями и инженерами без дополнительных устных пояснений.
